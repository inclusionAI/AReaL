# Custom Reward Functions

This guide explains how to create custom reward functions for RL training in AReaL.

## Reward Function Signature

Every reward function follows the same signature:

```python
def my_reward_fn(
    prompt: str,
    completions: str,
    prompt_ids: list[int],
    completion_ids: list[int],
    **kwargs,
) -> float:
    """
    Parameters
    ----------
    prompt : str
        The decoded prompt text.
    completions : str
        The decoded completion text generated by the model.
    prompt_ids : list[int]
        Token IDs of the prompt.
    completion_ids : list[int]
        Token IDs of the completion.
    **kwargs
        Additional fields from the dataset (e.g., "answer", "solution").

    Returns
    -------
    float
        A scalar reward value.
    """
    ...
```

The `**kwargs` receive all extra columns from your dataset. For example, if your dataset
has an `answer` field, it will be available as `kwargs["answer"]`.

## Rule-Based Rewards

The simplest approach is a rule-based reward that checks the model output against a
known answer.

### Example: Exact Match

```python
def exact_match_reward_fn(prompt, completions, prompt_ids, completion_ids, answer, **kwargs):
    return 1.0 if answer.strip() in completions else 0.0
```

### Example: Math Verification

AReaL includes a built-in math reward using the `math-verify` library:

```python
# areal/reward/gsm8k.py
from areal.reward import get_math_verify_worker

def gsm8k_reward_fn(prompt, completions, prompt_ids, completion_ids, answer, **kwargs):
    try:
        worker = get_math_verify_worker()
        return worker.verify(str(completions), str(answer))
    except Exception:
        return 0.0
```

## Model-Based Rewards (Critic-Like)

For tasks without clear-cut answers, you can use a **pretrained reward model** to score
completions. A typical critic-like reward model is a sequence classification model
(e.g., a Bradley-Terry model trained with
[`examples/alignment/`](https://github.com/inclusionAI/AReaL/tree/main/examples/alignment/))
that takes (prompt, response) and outputs a scalar score.

### Example: Using a Pretrained Reward Model

```python
import torch
from transformers import AutoModelForSequenceClassification, AutoTokenizer

_model = None
_tokenizer = None

def my_reward_model_fn(prompt, completions, prompt_ids, completion_ids, **kwargs):
    global _model, _tokenizer
    if _model is None:
        _tokenizer = AutoTokenizer.from_pretrained("my-org/my-reward-model")
        _model = AutoModelForSequenceClassification.from_pretrained(
            "my-org/my-reward-model", torch_dtype=torch.bfloat16
        ).cuda().eval()

    inputs = _tokenizer(
        prompt + completions,
        return_tensors="pt",
        truncation=True,
        max_length=2048,
    ).to("cuda")

    with torch.no_grad():
        score = _model(**inputs).logits.squeeze().float().item()
    return score
```

**Key points:**

- **Lazy loading**: Load the model once and cache it globally. The reward function is
  called many times per training step, so avoid reloading on each call.
- **GPU placement**: Use `torch.bfloat16` and `.cuda()` for efficiency. The reward model
  runs on the same GPU as the rollout worker.
- **Error handling**: Wrap the function body in a try/except and return 0.0 on failure
  to avoid crashing the training loop.

## Generative Rewards (LLM-as-Judge)

For tasks where correctness is hard to verify programmatically, you can use a
**generative model as a judge** — a separate LLM that evaluates the policy model's
output and produces a reward score.

AReaL includes a complete LLM-as-judge implementation in the
[Tongyi DeepResearch search agent example](https://github.com/inclusionAI/AReaL/tree/main/examples/search_agent/tongyi_deepresearch/).
The key pattern involves three components:

### 1. Judge Prompt Template

The judge receives the question, ground truth, and model prediction, then outputs a
structured JSON verdict:

````python
# From examples/search_agent/tongyi_deepresearch/react_agent.py
judge_prompt_template = (
    "You are an evaluation assistant. Please determine if the predicted answer "
    "is equivalent to the labeled answer.\n"
    "question: {question}\n"
    "ground truth answers: {gt_answer}\n"
    "pred_answer: {pred_answer}\n\n"
    "The output should in the following json format:\n"
    '```json\n{{\n'
    '    "rationale": "your rationale",\n'
    '    "judgement": "correct or incorrect"\n'
    '}}\n```\n'
)
````

### 2. Parse the Judge Response

````python
# From examples/search_agent/tongyi_deepresearch/react_agent.py
import ast, json

def parse_judge_result(raw_response):
    mbe = None
    for parse_fn in [json.loads, ast.literal_eval]:
        try:
            mbe = parse_fn(
                raw_response.split("```json")[-1].split("```")[0].strip()
            )
            break
        except Exception:
            pass
    if mbe is None and '"judgement": "correct"' in raw_response:
        mbe = dict(judgement="correct")
    if mbe is None:
        mbe = dict(judgement="unknown")
    return float("judgement" in mbe and mbe["judgement"] == "correct")
````

### 3. Call the Judge via a Separate Inference Engine

Unlike critic-like rewards, generative judges need their own inference engine. This is
done inside a custom workflow rather than as a standalone `reward_fn`:

```python
# From examples/search_agent/tongyi_deepresearch/train.py
class MyJudgeWorkflow(RolloutWorkflow):
    def __init__(self, ..., judge_engine_config=None):
        # Create a separate inference engine for the judge
        if judge_engine_config is not None:
            self.judge_engine = RemoteSGLangEngine(judge_engine_config)
            self.judge_engine.initialize()
        self.judge_client = ArealOpenAI(
            engine=self.judge_engine, tokenizer=tokenizer
        )
```

The judge is called asynchronously during episode collection:

```python
async def calc_reward_with_llm_judge(self, result):
    judge_prompt = judge_prompt_template.format(
        question=result["question"],
        gt_answer=str(result["answer"]),
        pred_answer=result["prediction"][:200],
    )
    completion = await self.judge_client.chat.completions.create(
        messages=[{"role": "user", "content": judge_prompt}],
        temperature=1.0,
        max_completion_tokens=8192,
        store=False,  # Don't record judge tokens as training data
    )
    return parse_judge_result(completion.choices[0].message.content)
```

**Why a separate engine?** Generative judges require GPU inference, which cannot run
inside the `ProcessPoolExecutor` used by `AsyncRewardWrapper`. Instead, they run as part
of a custom workflow with a dedicated inference engine.

For a complete working example, see
[`examples/search_agent/tongyi_deepresearch/`](https://github.com/inclusionAI/AReaL/tree/main/examples/search_agent/tongyi_deepresearch/).

## Async Reward Wrapper

For reward functions that may be slow (e.g., calling external APIs, running heavy
computation), AReaL provides `AsyncRewardWrapper` to run them in a process pool with
timeout handling:

```python
from areal.api.reward_api import AsyncRewardWrapper

# Wrap a sync reward function for async execution
async_reward = AsyncRewardWrapper(
    my_reward_fn,
    timeout_seconds=15,   # Return 0.0 if computation exceeds 15s
    max_workers=4,        # Process pool size
    max_retries=3,        # Retry on failure
)

# Use in workflow:
reward = await async_reward(prompt, completions, prompt_ids, completion_ids, **data)
```

The built-in `RLVRWorkflow` automatically wraps your reward function with
`AsyncRewardWrapper`.

## Integration with Training

Pass your reward function to the trainer as a string import path:

```python
from areal import PPOTrainer

with PPOTrainer(config, train_dataset=train_dataset) as trainer:
    trainer.train(
        workflow="areal.workflow.rlvr.RLVRWorkflow",
        workflow_kwargs=dict(
            reward_fn="my_module.my_reward_fn",  # String import path
            gconfig=config.gconfig,
            tokenizer=config.tokenizer_path,
        ),
    )
```

The string import path allows the reward function to be serialized and distributed to
worker processes.
